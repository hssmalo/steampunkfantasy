# Curation is a decoupled generate → promote flow through a gitignored candidates store

ADR 0006 settled *that* Assets are generated by AI, curated by a human, and
committed. This ADR settles the **mechanism**: curation is two decoupled CLI
steps, not one interactive command. `generate` writes N **Candidates** to a
gitignored `candidates/<race>/…` store (mirroring the `assets/` layout); a human
reviews them in whatever tool suits the kind (image viewer, editor, mesh
viewer); then a separate, kind-agnostic `promote <race> <kind> <name> --pick n`
copies the chosen Candidate into the committed `assets/` store.

**Why:** the human gate for an image or a 3D mesh means *opening the file*, not
reading a terminal prompt, so an interactive "pick 1/2/3" flow serves the review
badly and is near-impossible to test. Decoupling makes each half pure and
hermetically testable (no provider, no TTY), lets review happen out-of-band on
the human's schedule, and keeps the paid, non-deterministic AI call (`generate`)
cleanly separated from the deterministic file copy (`promote`).

## Consequences

- A **third top-level store** appears beside `assets/` (committed) and `output/`
  (gitignored Renderings): `candidates/` (gitignored, `config.paths.candidates`).
  The three-way split — Candidate vs Asset vs Rendering — is deliberate and each
  is named in `CONTEXT.md`.
- **`promote` is kind-agnostic and shipped by the foundation**; it needs no
  `Service`, only a Kind's `subdir`/`extension` to locate files. The per-kind
  `generate` subcommands (which need a real provider) land in the child issues.
- `promote` **overwrites an existing Asset silently** — git diff is the audit
  trail, consistent with how Renderings overwrite (ADR 0005). It does not
  auto-delete the sibling Candidates; the store is gitignored scratch.
- Rejected: an interactive single-command generate-and-pick flow — better UX for
  pure text, but it couples generation to review, fights image/mesh inspection,
  and resists testing.
